# Refining Response 3

This analysis is based on the fresh conversation in `messages`.

The system is improving, but the remaining failures are now mostly about:

- Prompt contracts that are still too loose for Qwen3:8B.
- Classifier routing for follow-up questions.
- Missing response validation/post-processing.
- Too much freedom in French-side explanations.

---

## Message 69 -> 70

Input:

```text
is this grammatically correct? -> J'habite en rue Dumas
```

Routing:

```text
intent: french_sentence
matched_rule: french_sentence_validation_request
input_excerpt: J'habite en rue Dumas
prompt_builder: Prompts::FrenchSentence
```

The routing is correct.

### What improved

The model correctly says:

```text
J'habite en rue Dumas
```

is incorrect.

It correctly suggests:

```text
J'habite rue Dumas.
```

### Problems

#### 1. Bad apostrophe in corrected sentence

In French output, it wrote:

```text
J'habite rue D'umas
```

That is wrong. `Dumas` is a proper noun and should not be split.

Expected:

```text
J'habite rue Dumas.
```

#### 2. Incorrect grammar explanation

It says:

```text
The correct preposition is 'rue' itself.
```

This is conceptually wrong. `rue` is not a preposition. It is a noun used in the
address expression.

Better:

```text
In French, with many street addresses, habiter can be followed directly by the street name:
J'habite rue Dumas.
The word rue is a noun, not a preposition.
```

#### 3. Redundant translation explanation

It wrote:

```text
Translation: I live on Dumas Street.
Translation explanation: I live on Dumas Street.
```

The second line adds no value.

#### 4. French-side self-glosses still appear

Examples:

```text
« habiter » (habiter)
« rue » (rue)
```

These are exactly the patterns we wanted to avoid.

#### 5. Synonyms are not useful

It wrote:

```text
habiter — loger
rue — avenue
```

This is not useful in this context.

`loger` is not a clean beginner synonym for `habiter` in every context.
`avenue` is not a synonym of `rue`; it is a different type of road.

### Suggested prompt improvements

For `Prompts::FrenchSentence`:

1. Remove the optional `# Synonyms` section for sentence validation.
2. Add a rule:

```text
Do not include synonyms unless the learner explicitly asks for synonyms.
```

3. Add a rule:

```text
Do not call nouns, articles, adjectives, or street-name elements "prepositions".
```

4. Add a correction guard:

```text
Never alter capitalization or apostrophes inside proper nouns unless the learner's error is specifically inside the proper noun.
For example, Dumas must stay Dumas, not D'umas.
```

5. Replace `Translation explanation` with a more useful section:

```text
# Natural Translation

# Correction

# Why

# Grammar Breakdown

# Notes
```

6. Tell the model not to repeat the translation as its own explanation.

## Message 71 -> 72

Input:

```text
validate C'est la correct rue?
```

Routing:

```text
intent: french_sentence
matched_rule: french_sentence_validation_request
input_excerpt: C'est la correct rue?
prompt_builder: Prompts::FrenchSentence
```

The routing is correct.

### What improved

The model correctly identified that the sentence is incorrect.

The proposed correction:

```text
C'est la rue correcte ?
```

is grammatically possible.

### Problems

#### 1. Duplicate correction

It wrote:

```text
The correct sentence is 'C'est la rue correcte?' or 'C'est la rue correcte?'
```

This is duplicate output.

#### 2. It ignores a more natural phrasing

For a beginner asking "is this the correct street?", a more natural French
sentence may be:

```text
C'est la bonne rue ?
```

or:

```text
Est-ce la bonne rue ?
```

`correcte` is grammatical, but `bonne` is often more idiomatic in this exact
context.

The tutor should distinguish:

```text
Grammatically correct:
C'est la rue correcte ?

More natural:
C'est la bonne rue ?
```

#### 3. Incorrect or awkward explanation

It says:

```text
'correct' is not used as an adjective in French to describe a street.
```

The real issue is that `correct` is English, while the French adjective is
`correct/correcte`, and French adjective placement/agreement matters.

Better:

```text
The word correct is English here. In French, the adjective is correct/correcte.
Because rue is feminine, use correcte.
Also, this adjective normally goes after the noun: la rue correcte.
```

#### 4. Bad decomposition

It says:

```text
C'est — contraction of ce est
```

For beginners, it is better to write:

```text
C'est — contraction of ce + est
```

In French:

```text
C'est — contraction de ce + est
```

#### 5. French-side self-glosses still appear

Examples:

```text
(rue)
(correct)
(il est)
```

These should not appear in `target_language`.

#### 6. Synonyms are weak again

```text
correcte — juste
rue — avenue
```

`rue — avenue` is especially misleading.

### Suggested prompt improvements

For sentence validation:

1. Ask for exactly one corrected sentence unless alternatives are meaningfully different.
2. Add a separate optional line:

```text
More natural:
...
```

3. Add:

```text
If the grammatically corrected sentence is less idiomatic than another common form, include both:
- Corrected sentence
- More natural sentence
```

4. Add:

```text
Do not list duplicate alternatives.
```

5. Remove synonyms by default.

## Message 73 -> 74

Input:

```text
straight en français
```

Routing:

```text
intent: vocabulary
lookup_mode: single_word
input_excerpt: straight
prompt_builder: Prompts::Vocabulary
```

The routing is correct.

### What improved

The model recognizes that `straight` has several meanings.

It mentions context.

It includes:

- shape/direction
- direct/honest
- immediate
- sexual orientation

This is much better than the previous response.

### Critical problem

The raw model response used the wrong JSON key:

```json
"french_language": "..."
```

instead of:

```json
"target_language": "..."
```

The app could not parse it and displayed the fallback message.

This is not only a model capability issue. The current prompt schema is probably
too weak for Qwen3:8B when the prompt gets longer.

### Other quality problems in the raw response

#### 1. The French explanation still starts from the French translation

It wrote:

```text
Le mot droit en français...
```

But the learner asked about the English word `straight`.

Better:

```text
Le mot anglais straight a plusieurs traductions en français.
```

#### 2. Some meaning categories are wrong

It says:

```text
straight = immediate or not delayed
```

Example:

```text
a straight answer
```

This is not "immediate"; it is more like direct/frank.

For "go straight home", the meaning is "directly/immediately":

```text
rentrer directement à la maison
```

#### 3. Sexual orientation phrasing is awkward

It wrote:

```text
a group of people with similar sexual orientations
```

Better:

```text
heterosexual
```

French:

```text
hétérosexuel / hétérosexuelle
```

or informally:

```text
hétéro
```

#### 4. It lacks the key beginner distinction

The user later asked:

```text
wasnt droit the translation for right?
```

The answer should proactively explain:

```text
droit can mean straight/right depending on context.
droite can be the feminine adjective straight/right, or the noun meaning "right side".
à droite means "to/on the right".
tout droit means "straight ahead".
```

### Suggested prompt improvements

For `VocabularyPrompt` single-word mode:

1. Use a stricter JSON schema block:

```text
Return exactly this JSON shape:
{
  "default_language": "...",
  "target_language": "..."
}

The property names must be exactly:
- default_language
- target_language

Do not use property names like french_language, english_language, source_language, or translation.
```

2. Put the schema instruction at the end of the prompt too, not only in the base
prompt. Small models often follow the last instruction better.

3. For single English words, include this structure:

```text
# Word

# Short Answer

# Common Meanings

# Grammar

# Examples

# Common Confusions
```

4. Add this instruction:

```text
If the English word maps to a French word that also has another common English meaning, explain the confusion.
Example: droit can mean straight, right, law, or rights depending on context.
```

5. Add this instruction:

```text
For direction, prefer:
- tout droit = straight ahead
- aller tout droit = to go straight ahead
- continuer tout droit = to keep going straight

For right side, prefer:
- à droite = to/on the right
```

## Message 75 -> 76

Input:

```text
wasnt droit the translation for right? what about straight as in going straight in the street
```

Routing:

```text
intent: english_sentence
matched_rule: english_statement
prompt_builder: Prompts::EnglishSentence
```

This routing is wrong.

The learner is asking a follow-up vocabulary/usage question, not submitting an
English sentence for translation.

### Problems in the answer

#### 1. Wrong language field behavior

Because it used `EnglishSentencePrompt`, `default_language` was empty.

But this was a vocabulary/usage explanation, so `default_language` should have
contained the English explanation, and `target_language` should have contained
the French version of the same explanation.

#### 2. Answer is only in French

The user likely needs the explanation in English first.

#### 3. Still has English parentheticals in French output

Examples:

```text
(pour « straight ahead »)
(He walked straight.)
(Turn right.)
```

These should not be in `target_language`.

#### 4. Some French is unnatural

It suggests:

```text
aller droit là-bas
en direction directe
```

For a street-direction context, better beginner phrases are:

```text
Va tout droit.
Continue tout droit.
Allez tout droit.
Marchez tout droit.
```

### Suggested classifier improvements

Add follow-up vocabulary/usage detection before `english_sentence`.

Return `:vocabulary` or a new intent `:usage` when the message:

- contains `what about`
- contains `as in`
- contains `wasn't`
- contains `wasnt`
- contains `doesn't`
- contains `doesnt`
- asks about a previously discussed word
- contains a quoted or known vocabulary word from recent conversation

Examples:

```text
wasnt droit the translation for right?
what about straight as in going straight in the street?
straight as in go straight ahead
does droit also mean right?
```

These should not go to `EnglishSentencePrompt`.

### Suggested architecture improvement

Add a new prompt builder:

```text
Prompts::Usage
```

or split vocabulary into:

```text
Prompts::Vocabulary::Lookup
Prompts::Vocabulary::Usage
```

The current `VocabularyPrompt` can answer lookup questions, but follow-up usage
questions need a different shape:

```text
# Short Answer

# Difference

# Use This For...

# Examples

# Common Mistake
```

For the user question about `droit` and `straight`, the desired English answer
would be something like:

```markdown
# Short answer

Yes. **droit** can mean both "straight" and "right", depending on context.

# Direction

For "go straight" in the street, use:

- Va tout droit.
- Continue tout droit.
- Allez tout droit.

# Right side

For "right" as the opposite of left, use:

- à droite
- tourner à droite

# Important distinction

- droit / droite = straight, right, correct, law/rights depending on context
- tout droit = straight ahead
- à droite = to/on the right
```

And the French answer should express the same content in French without English
parentheticals.

## Cross-cutting Recommendations

### 1. Add response key repair for common JSON mistakes

Qwen3:8B produced:

```json
"french_language"
```

instead of:

```json
"target_language"
```

Before falling back, the parser could repair known aliases:

```ruby
target_language = inner_payload["target_language"] ||
  inner_payload["french_language"] ||
  inner_payload["target"] ||
  inner_payload["french"]
```

Similarly:

```ruby
default_language = inner_payload["default_language"] ||
  inner_payload["english_language"] ||
  inner_payload["default"] ||
  inner_payload["english"]
```

This is pragmatic for local small models.

Keep the raw response stored so the bug is still debuggable.

### 2. Persist prompt text for debugging

Current metadata stores classifier and builder, but not the exact prompt.

For prompt refinement, add one of:

```text
prompt_text
```

or:

```text
prompt_digest
prompt_preview
```

Storing the full prompt may be useful during development. If it is too large,
store a digest and a truncated preview.

### 3. Add output validation warnings

After parsing, inspect the generated strings for known bad patterns:

- `D'umas`
- `rue (rue)`
- `Je (je)`
- `correcte (correcte)`
- same-language arrows like `ligne droite → ligne droite`
- `french_language` key in raw response
- duplicate alternatives: `X or X`

Store warnings in `prompt_metadata`.

This would let the UI show:

```text
Output warning: target_language contains same-language glosses.
```

### 4. Reduce optional sections

The current prompt asks for many sections:

- Translation
- Translation explanation
- Grammar validation
- Grammar breakdown
- Notes
- Synonyms

For Qwen3:8B this increases error rate.

Prefer fewer required sections:

```text
# Correction

# Translation

# Why

# Breakdown
```

Only include notes/synonyms when explicitly requested.

### 5. Make French target output simpler

The French version currently tries to mirror the English explanation too closely.
This leads to unnatural French and self-glosses.

Better instruction:

```text
The French answer should teach the same points, but it does not need to be a sentence-by-sentence translation of the English answer.
Write natural French for a beginner.
```

### 6. Add examples directly in the prompt

Small models follow examples better than abstract rules.

For French target output, include a bad/good example:

```text
Bad:
rue (rue)
une ligne droite → une ligne droite

Good:
rue — nom féminin
Exemple : une ligne droite.
```

### 7. Consider model limitations

Some errors are prompt-fixable:

- wrong routing
- wrong JSON key
- self-glosses
- duplicate sections

Some errors may still happen with Qwen3:8B:

- subtle idiomatic distinction between `correcte` and `bonne`
- malformed proper nouns like `D'umas`
- overconfident grammar claims

If this app depends on high-quality grammar correction, consider testing the same
prompts against a stronger model for comparison. But the current prompt can
still be improved substantially before switching models.
